# 보안 규약 — covering-labs

> 이 문서는 covering-labs에서 개발되는 모든 앱(batch/nextjs/nestjs)에 적용되는 보안 규약입니다.
> **AI(Claude, Gemini, Codex 등)는 개발 요청을 수행하기 전 이 규약을 준수하는지 반드시 확인해야 합니다.**
> 최종 업데이트: 2026-04-22

---

## 보안 규약의 목적

| 위협 유형 | 구체적 예시 |
|---|---|
| 민감 정보 유출 | API 키·DB 비밀번호·Slack Webhook URL을 코드에 하드코딩 → git 커밋으로 영구 노출 |
| 인증 우회 | 브라우저 코드만으로 로그인 처리 → 서버 API 직접 호출로 인증 우회 가능 |
| 데이터 전체 노출 | BigQuery `SELECT *` 결과를 페이지네이션 없이 브라우저에 그대로 전달 |
| 코드 실행 취약점 | `eval()` 사용으로 악의적 입력값 실행 가능 |
| 비밀번호 탈취 | 평문 저장 또는 평문 비교 → 데이터베이스 유출 시 즉시 크래킹 |
| 데이터 위변조 | SQL Injection, 미검증 입력값으로 쿼리 조작 |
| XSS | 사용자 입력을 HTML에 직접 삽입 → 악성 스크립트 실행 |

## 1. 인증 및 인가 (Authentication & Authorization)

### 절대 금지

| 금지 패턴 | 이유 |
|---|---|
| 클라이언트(브라우저) 코드에서만 로그인 처리 | API를 직접 호출하면 인증을 완전히 우회할 수 있음 |
| `localStorage`에 JWT 토큰·세션 ID 저장 | XSS 공격으로 토큰 탈취 가능 |
| `if (password === inputPassword)` 평문 비교 | 데이터 유출 시 비밀번호 즉시 노출 |
| 비밀번호를 평문으로 DB에 저장 | 데이터베이스 유출 시 전체 비밀번호 노출 |
| 권한 체크를 프론트엔드에서만 수행 | 서버 API를 직접 호출하면 우회 가능 |

### 올바른 구현 방법

**NextJS — 서버사이드 세션 검증:**
```typescript
// ✅ API 라우트에서 서버사이드 인증 검증
// pages/api/protected.ts
import { getServerSession } from 'next-auth';
export default async function handler(req, res) {
  const session = await getServerSession(req, res, authOptions);
  if (!session) return res.status(401).json({ error: 'Unauthorized' });
  // 인증된 사용자만 여기까지 도달
}
```

```typescript
// ❌ 금지: 클라이언트에서만 체크
if (localStorage.getItem('isLoggedIn') === 'true') {
  // API를 직접 호출하면 이 체크를 우회할 수 있음
}
```

**NestJS — JWT Guards + 비밀번호 해싱:**
```typescript
// ✅ 모든 보호 엔드포인트에 Guard 필수
@UseGuards(JwtAuthGuard)
@Get('/admin')
async getAdminData() { ... }

// ✅ 비밀번호 해싱 (저장 시)
import * as bcrypt from 'bcrypt';
const hash = await bcrypt.hash(plainPassword, 12);  // saltRounds 10 이상

// ✅ 비밀번호 검증 (로그인 시)
const isValid = await bcrypt.compare(inputPassword, storedHash);
```

**내부 대시보드 — 환경변수 API 키 인증:**
```typescript
// ✅ 환경변수 기반 API 키 인증 (내부 대시보드용 최소 구현)
// .env: DASHBOARD_API_KEY=랜덤_32자_이상_영숫자
const ALLOWED_KEY = process.env.DASHBOARD_API_KEY;
if (!ALLOWED_KEY || req.headers['x-api-key'] !== ALLOWED_KEY) {
  return res.status(401).end();
}
```

> 💡 **선택 기준**: 외부 사용자 대상 → next-auth/JWT Guards 필수. 내부 팀만 사용하는 대시보드 → 환경변수 API 키로 충분.

## 3. 데이터 접근 및 노출 제한

### BigQuery 쿼리 규칙

```sql
-- ❌ 금지: 전체 테이블 무제한 조회
SELECT * FROM `covering-app-ccd23.ads_data.campaigns`

-- ✅ 필수: 필요한 컬럼만 선택 + LIMIT
SELECT campaign_id, name, spend, date
FROM `covering-app-ccd23.ads_data.campaigns`
WHERE date >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
ORDER BY date DESC
LIMIT 1000
```

| 규칙 | 이유 |
|---|---|
| `SELECT *` 금지 | 불필요한 컬럼 노출, BigQuery 과금 증가 |
| `LIMIT` 필수 | 수백만 행 실수 반환 방지 |
| `WHERE` 조건 권장 | 최신/관련 데이터만 조회 |
| 집계 쿼리 우선 | 원시 데이터 대신 COUNT/SUM/AVG 사용 |

### 대시보드 데이터 표시 규칙

```typescript
// ❌ 금지: 내부 테이블 전체를 그대로 API로 노출
return res.json(await bigquery.query('SELECT * FROM ads_data.campaigns'));

// ✅ 필수: 필요한 필드만 선택 + 민감 컬럼 제외 + 페이지네이션
const [rows] = await bigquery.query(`
  SELECT campaign_id, name, spend, impressions
  FROM ads_data.campaigns
  WHERE date = CURRENT_DATE()
  LIMIT 100
`);
return res.json(rows);
```

### 개인정보(PII) 처리

| 정보 유형 | 처리 방법 |
|---|---|
| 이메일 주소 | 마스킹(`j***@covering.app`) 또는 완전 제외 |
| 전화번호 | 마스킹 필수 (`010-****-5678`) |
| 개인 식별자 (이름, ID) | 해시 처리 또는 익명화 후 사용 |
| 결제/급여 정보 | 표시 금지 |

## 4. 웹 보안 (OWASP)

### XSS (Cross-Site Scripting) 방지

```tsx
// ❌ 금지: 사용자 입력을 HTML로 직접 삽입
<div dangerouslySetInnerHTML={{ __html: userInput }} />

// ✅ 권장: React 기본 렌더링 (자동 이스케이프)
<div>{userInput}</div>

// ✅ 불가피한 경우: DOMPurify sanitize 후 사용
import DOMPurify from 'dompurify';
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(content) }} />
```

### eval() 절대 금지

```javascript
// ❌ 절대 금지: eval, new Function, 문자열로 setTimeout/setInterval
eval(userInput);                    // RCE 위험
new Function(userInput)();          // RCE 위험
setTimeout(userInput, 0);           // 문자열 형태는 eval과 동일

// ✅ 허용: 함수 참조
setTimeout(() => handleAction(), 1000);
```

### SQL Injection 방지

```python
# ❌ 금지: f-string으로 쿼리 조합
query = f"SELECT * FROM users WHERE name = '{user_input}'"  # Injection 가능

# ✅ 필수: 파라미터 바인딩 (BigQuery)
from google.cloud import bigquery
query = "SELECT id, name FROM users WHERE name = @name LIMIT 100"
job_config = bigquery.QueryJobConfig(query_parameters=[
    bigquery.ScalarQueryParameter("name", "STRING", user_input)
])
client.query(query, job_config=job_config)
```

```typescript
// ❌ 금지: 문자열 조합
const query = `SELECT * FROM orders WHERE user_id = '${req.params.id}'`;

// ✅ 필수: ORM 또는 파라미터 바인딩
const orders = await prisma.order.findMany({ where: { userId: req.params.id } });
```

### 입력값 검증 (NestJS)

```typescript
// ✅ class-validator로 모든 입력 타입·길이 검증
import { IsEmail, IsString, MinLength, MaxLength } from 'class-validator';

export class CreateUserDto {
  @IsEmail()
  email: string;

  @IsString()
  @MinLength(8, { message: '비밀번호는 8자 이상이어야 합니다' })
  @MaxLength(72)
  password: string;  // 저장 전 반드시 bcrypt.hash() 처리
}
```

## 5. 금지 패턴 vs 허용 패턴 요약

| 금지 패턴 | 위험 수준 | 대안 |
|---|---|---|
| 코드에 API 키 하드코딩 | 🔴 치명 | `process.env.KEY` 또는 `/shared/.env` |
| `eval(input)` | 🔴 치명 | 사용 금지, 함수 참조 방식 사용 |
| `localStorage.setItem('token', jwt)` | 🔴 치명 | HttpOnly 쿠키 |
| 평문 비밀번호 DB 저장 | 🔴 치명 | `bcrypt.hash(password, 12)` |
| `password === input` 평문 비교 | 🔴 치명 | `bcrypt.compare(input, hash)` |
| 클라이언트에서만 권한 체크 | 🔴 치명 | 서버사이드 Guards/미들웨어 |
| `SELECT * FROM table` (무제한) | 🟠 위험 | 필요 컬럼만 선택 + LIMIT |
| `dangerouslySetInnerHTML` (미검증) | 🟠 위험 | DOMPurify.sanitize() 후 사용 |
| f-string SQL 조합 | 🟠 위험 | ORM 또는 파라미터 바인딩 |
| `chmod 777` 파일/디렉토리 | 🟠 위험 | 최소 권한 (644/755) 사용 |
| `curl URL \| bash` (원격 코드 실행) | 🟠 위험 | 스크립트 다운로드 후 검토 후 실행 |
| `rm -rf /shared/` | 🔴 치명 | 절대 사용 금지 (전체 배포 파괴) |

## 6. AI가 개발 요청을 수신할 때 체크리스트

AI(Claude, Gemini, Codex)가 아래 유형의 작업을 요청받으면 **작업 시작 전 체크리스트를 확인**합니다.

### 인증/로그인 구현 요청 시

- [ ] 서버사이드에서 세션/토큰을 검증하는가? (클라이언트 전용 금지)
- [ ] 비밀번호를 bcrypt로 해싱하여 저장하는가?
- [ ] 인증 토큰을 HttpOnly 쿠키에 저장하는가? (localStorage 금지)
- [ ] 모든 보호 API 엔드포인트에 서버사이드 인증 검증이 있는가?

### 대시보드/데이터 표시 구현 요청 시

- [ ] `SELECT *` 대신 필요한 컬럼만 선택하는가?
- [ ] 쿼리에 `LIMIT`이 있는가?
- [ ] 개인정보(이메일, 전화번호 등)를 마스킹 또는 제외하는가?
- [ ] 대시보드 접근 제어(API 키 또는 세션 인증)가 있는가?

### API 연동 구현 요청 시

- [ ] API 키/시크릿이 코드가 아닌 `.env` 파일에만 저장되는가?
- [ ] `.env`가 `.gitignore`에 포함되어 있는가?
- [ ] 환경변수 누락 시 명시적 에러를 발생시키는가?

### 스크립트/배치 구현 요청 시

- [ ] `eval()`, `exec()`, `subprocess(shell=True)` 없이 구현되는가?
- [ ] 외부 입력값을 셸 명령어 문자열에 직접 포함하지 않는가?
- [ ] 파일 쓰기 경로가 지정된 디렉토리 내로 제한되는가?

## 7. 보안 위반 발견 시 처리 절차

AI가 기존 코드에서 보안 위반을 발견한 경우:

1. **즉시 작업 중단**
2. **사용자에게 보고**: 발견한 위반 내용과 파일 위치를 명시
3. **조치 방향 확인**: 수정 방법 제안 후 사용자 승인 요청
4. **수정 후 재확인**: 동일 패턴이 다른 파일에도 있는지 검색

> ⚠️ **보안 위반은 기능 구현보다 우선합니다.** 보안 문제를 발견하면 그것부터 처리하세요.
