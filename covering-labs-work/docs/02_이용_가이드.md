# 이용 가이드

> 대상: 모든 `@covering.app` 구성원
> 코드를 서버에 올리고, 실행하고, 자동화하는 방법

---

## 자동 배포 (GitHub Actions) — 권장

> `covering-app/covering-labs` 레포의 `apps/` 디렉토리에 앱을 추가하고 push하면 자동으로 배포됩니다.

### 배포 흐름

```
앱 폴더 작성 → 브랜치 push → PR 생성 → CodeRabbit 리뷰 → 승인 후 머지
    → GitHub Actions 감지 → VM에 코드 복사 → 빌드 + 실행 → Slack 알림 (접속 주소)
```

### 1단계 — 앱 폴더 만들기

```
covering-labs/apps/
└── 내-앱-이름/          ← 폴더 이름 = URL 경로 = 앱 식별자 (변경 불가)
    ├── deploy.yml       ← 필수
    └── ...코드...
```

> 참고: `apps/_template/` 에 타입별 예시 코드가 있습니다.

### 2단계 — deploy.yml 작성

**배치 스크립트 (Python):**
```yaml
name: my-batch          # 앱 이름 (Slack 알림에 표시)
description: "30분마다 실행되는 데이터 수집 스크립트"
type: batch
schedule: "0 9 * * 1-5"
command: "python3 src/main.py"
```

**Next.js 서버:**
```yaml
name: my-nextjs
description: "영업 현황 모니터링 대시보드"
type: nextjs
```

**NestJS 서버:**
```yaml
name: my-nestjs
description: "데이터 처리 REST API 서버"
type: nestjs
```

### 3단계 — 브랜치 push 및 PR 생성

```bash
git add apps/내-앱-이름/
git commit -m "feat: 내 앱 추가"

# main 직접 push 금지 — 브랜치 + PR 필수
git checkout -b feat/$(date +%Y-%m-%d)-내-앱-이름
git push origin feat/$(date +%Y-%m-%d)-내-앱-이름
gh pr create --title "feat: 내 앱 추가" --body "변경 내용 설명"
```

### 배포 완료 시 Slack 알림

배포가 완료되면 `#개발팀_커버링랩스` 채널로 알림이 옵니다:

```
✅ my-nextjs 배포 완료
🌐 접속 주소: https://labs.covering.app/my-nextjs     # private 앱
🌐 접속 주소: https://public-labs.covering.app/my-nextjs  # public 앱

✅ my-batch 배포 완료   (batch는 private 전용)
⏰ 스케줄: 0 9 * * 1-5
```

| 앱 유형 | URL | 접근 조건 |
|---|---|---|
| **private** (`apps/private/[앱]/`) | `https://labs.covering.app/[앱]` | AWS Client VPN(`covering-vpn-v2`) 필수 (2026-04-17부터) |
| **public** (`apps/public/[앱]/`) | `https://public-labs.covering.app/[앱]` | VPN 불필요, 외부 접근 가능 |

### batch 앱 실행 방식

batch 앱은 배포 시 **서버의 SA 유저(`sa_109369409955768144646`) crontab**에 자동 등록됩니다. 개인 crontab에 등록되지 않습니다.

> `deploy.yml`의 `schedule` 값에 인라인 주석(`#`)을 쓰지 마세요. crontab 등록이 실패할 수 있습니다.
> ```yaml
> schedule: "0 9 * * 1-5"           # ✅ 올바름
> schedule: "0 9 * * 1-5"  # 평일   # ❌ 주석 금지
> ```

### 포트 자동 배정

- 서버 앱(Next.js, NestJS)은 3001번부터 순차적으로 포트가 자동 배정됩니다
- nginx가 `https://labs.covering.app/앱이름` (private) 또는 `https://public-labs.covering.app/앱이름` (public) → 내부 포트로 자동 라우팅
- 포트 번호는 직접 관리하지 않아도 됩니다

### 주의사항

- 폴더 이름이 URL 경로가 됩니다 (`apps/private/my-shop/` → `https://labs.covering.app/my-shop`)
- `apps/_template/` 은 예시용 — 수정하지 마세요
- cron 시간은 **서버 시간 기준 (KST)**

### 앱 제거

앱 폴더를 삭제하고 push하면 자동으로 서버에서 내려갑니다.

```bash
git rm -r apps/private/내-앱-이름/   # 또는 apps/public/내-앱-이름/
git commit -m "chore: 내-앱-이름 제거"

# main 직접 push 금지 — 브랜치 + PR 필수
git checkout -b fix/$(date +%Y-%m-%d)-remove-내-앱-이름
git push origin fix/$(date +%Y-%m-%d)-remove-내-앱-이름
gh pr create --title "chore: 내-앱-이름 제거" --body "앱 삭제"
```

자동 처리: PM2 중지/삭제 + nginx 설정 삭제 + 포트 반환 + 앱 파일 삭제 + Slack 알림

## 코드 가져오기

### GitHub에서 clone

각자 개인 GitHub 계정으로 사용. 처음 한 번만 인증하면 이후 자동.

**1. GitHub Personal Access Token 생성**

https://github.com/settings/tokens → "Generate new token (classic)" → `repo` 체크 → 생성

**2. 서버에서 clone**
```bash
git clone https://본인깃헙아이디:토큰@github.com/org/repo.git
```

**3. 토큰 저장 (매번 입력 안 하려면)**
```bash
git config --global credential.helper store
git pull   # 한 번만 입력하면 이후 자동
```

> 토큰은 `~/.git-credentials`에 저장. 본인 홈 디렉토리 안이므로 다른 사용자는 볼 수 없음.

### 로컬 파일 전송

```bash
# 로컬 → 서버
gcloud compute scp ./파일.txt \
  covering-labs-instance-20260306-050059:~/파일.txt \
  --zone=asia-northeast3-a --project=covering-app-ccd23

# 폴더 통째로
gcloud compute scp --recurse ./my-project/ \
  covering-labs-instance-20260306-050059:~/my-project/ \
  --zone=asia-northeast3-a --project=covering-app-ccd23

# 서버 → 로컬
gcloud compute scp \
  covering-labs-instance-20260306-050059:~/결과.txt ./결과.txt \
  --zone=asia-northeast3-a --project=covering-app-ccd23
```

## 실행 방법 선택

| | `crontab` (스케줄러) | `tmux` (세션 유지) |
|---|---|---|
| 역할 | 정해진 시간에 자동 실행 | SSH 끊어져도 프로세스 유지 |
| 예시 | "매일 오전 9시에 실행해줘" | "이거 실행해놓고 퇴근할게" |
| 서버 재시작 후 | 자동으로 다시 동작 | 세션 날아감 (다시 실행) |
| sudo 필요 | ❌ 누구나 가능 | ❌ 누구나 가능 |

**정기적으로 반복** → `crontab` / **한 번 실행하고 켜놓기** → `tmux`

## 정기 실행 등록하기 (crontab)

누구나 sudo 없이 가능. 사용자별 독립 관리 — 다른 사람 것은 볼 수 없음.

**1단계: 스크립트 작성**

```bash
mkdir -p ~/my-batch/logs
nano ~/my-batch/run.py
```

```python
# ~/my-batch/run.py
import gspread
import google.auth
from datetime import datetime

creds, _ = google.auth.default(scopes=[
    "https://www.googleapis.com/auth/spreadsheets",
    "https://www.googleapis.com/auth/drive",
])
gc = gspread.authorize(creds)
sheet = gc.open("시트 이름").sheet1

data = sheet.get_all_records()
print(f"[{datetime.now()}] {len(data)}건 처리 완료")
```

**2단계: 스케줄 등록**

```bash
crontab -e    # 편집기 열림 (처음이면 nano 선택)
```

```bash
# 매일 오전 9시
0 9 * * * cd ~/my-batch && python3 run.py >> logs/batch.log 2>&1

# 10분마다
*/10 * * * * cd ~/my-batch && python3 run.py >> logs/batch.log 2>&1

# 매주 월요일 오전 6시
0 6 * * 1 cd ~/my-batch && python3 run.py >> logs/batch.log 2>&1

# 평일(월~금) 오전 8시 30분
30 8 * * 1-5 cd ~/my-batch && python3 run.py >> logs/batch.log 2>&1
```

저장 후 종료하면 즉시 적용.

**3단계: 확인 / 관리**

```bash
crontab -l              # 등록된 스케줄 목록
tail -f ~/my-batch/logs/batch.log   # 로그 실시간 확인
crontab -r              # 전체 스케줄 삭제 (주의!)
```

**정기적으로 GitHub 코드 자동 pull:**
```bash
0 8 * * * cd ~/my-repo && git pull >> ~/logs/git-pull.log 2>&1
```

> **🔴 필수 규칙:**
> 1. crontab 시간은 서버 시간 기준 (KST)
> 2. 반드시 `>> logs/파일.log 2>&1`을 붙이세요. 안 붙이면 에러가 나도 알 수 없음
> 3. 로그 파일 디렉토리를 미리 만들어야 함 (`mkdir -p ~/logs`)

## SSH 끊어져도 프로세스 유지 (tmux)

```bash
tmux new -s 작업이름        # 새 세션 시작
python3 my_script.py       # 스크립트 실행
# Ctrl+B → D              # 빠져나옴 (프로세스 계속 실행)

tmux attach -t 작업이름      # 다시 접속
tmux ls                     # 실행 중인 세션 목록
tmux kill-session -t 작업이름  # 세션 종료
```

> ⚠️ 서버 재시작되면 tmux 세션은 사라짐. 정기 실행이 필요하면 crontab을 쓰세요.
