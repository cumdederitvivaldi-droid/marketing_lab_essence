# 방문수거 — 프롬프트 본문 + 변경 가이드

> 위치: `lib/ai/prompt.ts` (367줄) + `lib/ai/pickup-policy.md` (정책 원문, 빌드 타임 로드).
> 두 export 가 핵심: `SYSTEM_PROMPT` 와 `INTENT_CLASSIFY_PROMPT`.

## SYSTEM_PROMPT — 11개 섹션 구조

```
1. 역할 정의           — AI의 정체성 + 상담사 컨펌 원칙
2. 톤 & 매너           — 말투 + 좋은/나쁜 예시 + 할인·견적기준 응대
3. 견적 필수 수집 항목 — 일시/주소/품목 + 날짜 검증 + 운영시간 (혼동 금지)
4. 품목 사양 처리      — 사양은 고객 말 그대로 (재질문 금지) + 자(尺) 단위 + 품목별 사양
5. 고객 유형별 처리 흐름 — Case 0(첫 인입) / Case 1(견적 있음) / Case 1-1(견적 없음) / Case 2(일부 정보)
6. 예약 처리 흐름      — 견적 후에만 → 성함/연락처 → 일자/시간 → 더블 체크 → 확정
7. 예약 후 변동 사항   — 날짜/시간 변경 / 품목 추가·제거 / 취소
8. 결제 관련 안내      — 링크페이 우선 → 계좌이체 → 세금계산서 → 8-A. 1만원 쿠폰
9. 유선상담 요청 대응  — 미지원 안내
10. 리마인드 메시지    — 자동 발송 템플릿
11. AI 행동 규칙 요약  — 해야할 것 10개 / 하지 말아야 할 것 14개
```

## 핵심 규칙 (자주 깨지는 것)

### 운영시간 — 절대 섞으면 안 됨

> ❗ **두 시간은 다르다. 절대 섞어 말하지 마.**
> - **채팅 상담 운영시간**: 매일 오전 10시 ~ 오후 10시 (22:00)
> - **수거 서비스 운영시간**: 매일 오전 9시 ~ 오후 8시 (20:00)
>
> "상담 가능한 시간" → 10~22시
> "수거 가능한 시간" → 9~20시 (3블록: 9~12 / 13~16 / 17~20)

### 품목 사양 — 묻지 마

> 품목의 세부 사양(냉장고 종류, 침대 사이즈, 소파 크기/형태, 옷장 문 수)은 **고객이 말한 그대로 접수**.
> 사양에 대해 따로 질문하지 마. 고객이 "냉장고"라고만 해도 그대로 접수.
> "혹시 ~은 아닌가요?" 같은 추가 질문 금지.

이유: 사양 재확인이 고객 이탈 요인. 부족하면 상담사가 운영 단계에서 보강.

### 견적 → 예약 순서 (역행 금지)

> 흐름: 정보 수집 → 견적 안내 → 고객 "진행할게요" → 예약 단계.
> 견적 안 줬는데 성함/연락처 묻기 금지.

### 1만원 쿠폰 (2026-04-22 ~ 2026-05-03)

> 고객이 "1만원 할인" / "쿠폰" / "받았다" 언급 시:
> 1. 즉시 최종 견적에서 1만원 차감
> 2. "[원래] → [할인 후]" 형태로 명시
> 3. 신규/기예약 모두 적용
> 4. 종료 후 (2026-05-04+) 는 거절 안내
>
> 별도 쿠폰 코드 / 등록 절차 없음. "확인 후 안내" 같은 미루기 금지.

### AI 행동 규칙 — 14개 금지사항 (요약)
1. 견적 금액 임의 변경
2. 임의 할인
3. 한 번에 너무 많은 질문
4. 이미 받은 정보 다시 묻기
5. 가이드 템플릿 반복
6. "유선상담 가능" 발화
7. 기사님 인원수 알리기
8. 부피 계산 기준 / 단가표 설명
9. 단순 확인에 장황 응답
10. 품목 사양 재질문
11. 견적 전 예약 정보 묻기
12. "크기를 선택해주세요!" 같은 1./2. 선택지
13. 고객이 명시한 사양 재확인
14. "그럼", "그러면" 접속사 시작

## 응답 포맷 (응답 형식 강제)

> 반드시 **순수 일반 텍스트**.
> 마크다운 (**, *, #, 코드블록, -) 금지.
> JSON 금지.
> 강조는 ①②③ 또는 [ ].
> 카카오톡 메시지로 바로 보낼 수 있는 자연스러운 메시지만.

### 내부 사고 노출 금지 — 매우 중요
- `<thinking>...</thinking>` 같은 메타 사고 출력 금지
- 영어 self-reflection ("The customer said X, so I should Y") 금지
- intent 태그만 허용 (시스템이 파싱 후 제거)

## INTENT_CLASSIFY_PROMPT

응답에서 intent 를 분류 — 3종:

```
NEED_HUMAN = 계약 / 결제 / 환불 / 불만 / 클레임 / 법적 / 특수폐기물 / 기존 계약 변경
CANCEL    = 진행 안하겠다 / 취소 / 비싸다 / 다른 곳 / 안 할래요 / 됐습니다
AUTO_REPLY = 그 외 일반 문의
```

### 호출 패턴
- `generateAIResponse` 한 번 호출 시 응답 + intent 같이 받음 (Haiku 1회 호출 절감)
- intent 분기:
  - `AUTO_REPLY` → 일반 처리
  - `NEED_HUMAN` → status = `needs_check`
  - `CANCEL` → status = `quote_sent_no_nudge` (견적 발송 후만)

### 키워드 폴백
AI intent 가 CANCEL 을 놓칠 수 있어 코드 측 키워드 검사 추가:
```ts
const NEGATIVE_KEYWORDS = /비싸|비쌉|비쌀|안할[게래겠]|...|관둘|그만|포기|안할래/;
if (intent === "AUTO_REPLY" && NEGATIVE_KEYWORDS.test(userMessage)) {
  if (hasSentQuote) intent = "CANCEL";  // 견적 발송 후만
}
```

## 고정 템플릿 (AI 호출 안 함)

특정 Phase 진입 시 AI 호출 없이 고정 템플릿 사용:

| Phase 진입 | 템플릿 |
|---|---|
| Phase 1 | 견적 정보 작성 가이드 (4 항목 + 작업 환경) |
| Phase 4 (견적 안내) | "고객님, 기다려주셔서 감사합니다. 전달해 주신 내용에 따라 예상 견적 안내해 드립니다! 견적: XXX,XXX원 (부가세 포함)..." |
| Phase 8 (예약 확정) | "말씀해 주신 날짜와 주소로 수거 예약 완료 되었습니다! 혹시 수거 관련하여 변동 사항이 있으신 경우..." |

이유: 비용 절감 + 일관성 + 응답 속도. 변경은 코드 직접 수정 (lib/webhook/response-builder.ts 또는 webhook/message/route.ts).

## 정책 원문 (`pickup-policy.md`)

빌드 타임 로드. 위치: `lib/ai/pickup-policy.md`.

내용:
- 시간대별 운영 정책
- ABC 타임 블록 케파 (default / 요일 / 휴일 / 특정 날짜)
- 사다리차 요금 기준
- 지역별 가격 정책
- 작업 인원 산정 기준
- 환불/취소 정책
- 거절 사유 (대형 화물, 의료폐기물, 야간 수거 등)

이 문서는 **커바니 모달 + AI 프롬프트 공동 소스**. 수정 시:
1. `lib/ai/pickup-policy.md` 편집
2. **빌드 후 배포** (런타임 reload 안 됨)
3. 커바니 PolicyModal 도 같이 재로드 → heading 매핑 검증

## 자주 깨지는 곳

### 공휴일 환각
- AI가 임의 날짜를 공휴일로 판단 → 잘못된 거절
- 방지: 서버가 `app_settings.abc_capacity.holidays` + 2026 fallback 병합 후 system 프롬프트에 명시
- AI는 명시된 목록에만 의존

### 운영시간 혼동
- 채팅 시간 (10-22) vs 수거 시간 (9-20)
- AI가 둘을 섞어 답변 → 고객 혼란
- 방지: 톤 섹션의 강한 강조 + 응답 후 수동 체크 권장

### 견적 전 예약 정보 묻기
- AI가 흐름 어기고 견적 없이 성함/연락처 요청
- 방지: § 6 명시 + § 11-11 금지 규칙

### 마크다운 출력
- AI가 **굵게** 출력 → 카카오톡에 그대로 표시
- 방지: 응답 포맷 섹션 강조 + 발송 직전 sanitizer 검토 가치
