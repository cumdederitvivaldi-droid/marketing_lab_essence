# 03 — AI 로직 (인사이트 + 분류)

## 모델

| 용도 | 모델 | 위치 |
|---|---|---|
| Customer Journey 인사이트 | Sonnet (`claude-sonnet-4-6`) | `lib/dashboard/insight.ts` + `/api/new_dashboard/insight` |
| 상담사 리포트 (CS report) | Sonnet | `/api/new_dashboard/cs-report` |
| P5 넛지 이탈 사유 분류 | Haiku | `/api/new_dashboard/p5-reasons` + `lib/dashboard/p5-classify.ts` |
| Phase 2/4/5/8 이탈 사유 분류 | Haiku | `/api/new_dashboard/churn-reasons` + `lib/dashboard/churn-classify.ts` |
| 고객 불만 사전 분류 (5분 cron) | Haiku | `/api/cron/classify-complaints` + `lib/dashboard/complaint-classify.ts` |

대시보드는 Sonnet/Haiku 비용 절감을 위해 **모든 AI 호출을 캐시**.

## 캐시 전략

### 1. DB 캐시 — `dashboard_insights` (Sonnet)
Customer Journey AI 인사이트:
- 캐시 키: `(period_key, journey_hash)` 둘 다 같으면 hit
- `period_key` = `"thisMonth_2026-04-01_2026-04-24"` 같은 패턴
- `journey_hash` = journeyMap 핵심 데이터 sha256 16자 → 데이터 변화 없으면 hit
- TTL 없음 (hash 가 같은 한 영구)

### 2. DB 캐시 — `dashboard_p5_reasons` / `dashboard_churn_reasons` (Haiku)
이탈 사유 분류:
- session_id 별 1회 분류 후 캐시
- 같은 session 재요청 시 즉시 반환

### 3. DB 캐시 — `dashboard_complaints` / `complaint_classifications` (Haiku, cron)
- `cron/classify-complaints` (5분) 가 최근 7일 미분류 user 메시지 배치 분류
- (session_id, message_id) PK + UPSERT
- 대시보드 진입 시 캐시 hit 으로 즉시 표시 (cron 이 미리 채워둠)

### 4. 메모리 캐시 — `lib/dashboard/cache.ts`
- `csReportCache` — 상담사 리포트
- `churnReasonsCache` — 이탈 사유
- 프로세스 동안 유지 (Vercel function 재시작 시 사라짐)
- 클라이언트 prefetch 와 결합해 첫 로딩 즉각 표시

### 5. 클라이언트 prefetch — `lib/cache/prefetch.ts`
- `prefetchDashboardData` — 페이지 진입 시 background fetch
- 데이터 도착 후 `localStorage` + 메모리에 저장
- 모달 클릭 시 캐시 우선 read → 즉시 표시 (네트워크 없음)

## Customer Journey 인사이트 (Sonnet)

### 입력
- journeyMap 객체 (각 phase 의 진입자 / 전환율 / 이탈 keyword 등)
- 기간 (period_key)

### 호출
1. 클라이언트 → `/api/new_dashboard/insight` POST
2. 서버: hash 계산 → `dashboard_insights` 조회
3. hit → 즉시 반환
4. miss → Sonnet 호출 → INSERT → 반환

### 출력
운영자가 읽을 한국어 인사이트 본문 (markdown 가능):
- 가장 큰 이탈 지점 + 추정 사유
- KR 진척도 + 이번 달 마감까지 필요 페이스
- 권장 조치 (예: "Phase 4 견적→예약 전환율이 5pt 떨어짐, 가격 메시지 점검 필요")

## 상담사 리포트 (CS report — Sonnet)

`/api/new_dashboard/cs-report` 가 한 상담사의 일/주/월 활동 요약 + 강점·약점 + 개선 제안.

### 사용자 피드백 반영
- 모델: `claude-sonnet-4-6` (Haiku 가 아닌 Sonnet 사용)
- 약점은 **고객 불만 발생 시에만** 언급 (단순 비율로 약점 추정 금지)
- 개선 사항은 **0~4개 허용** (강제 4개 X)

### 캐시
- 메모리 + DB (구현 디테일은 코드 참조)
- 사용자 메모: 백그라운드 prefetch 추가 → 모달 즉시 표시

## P5 넛지 이탈 사유 분류 (Haiku)

P5 = 견적 발송 후 넛지 단계 (`status = quote_sent_nudge`). 그 단계에서 이탈한 고객의 이유를 분류:
- 가격 부담
- 일정 미정
- 단순 정보 수집
- 무응답
- 기타

분류는 on-demand (대시보드 진입 시) — Haiku 호출 후 `dashboard_p5_reasons` 캐시.

## Phase 이탈 사유 분류 (Churn)

Phase 2, 4, 5, 8 각각의 이탈 사유 — 동일 패턴.

`lib/dashboard/churn-classify.ts` + `dashboard_churn_reasons` 캐시.

## 불만 분류 (`classify-complaints` cron)

### 동작 (5분마다)
1. 최근 7일 messages (role=user, 6자+) 중 미분류 row 만 조회
2. Haiku 배치 분류 (BATCH_SIZE=30 × MAX_BATCHES=10 = 최대 300건/회)
3. 결과를 `dashboard_complaints` 또는 `complaint_classifications` 테이블에 UPSERT
4. PK: `(session_id, message_id)` — 중복 분류 방지

### 분류 결과 (m027~031 진화)
- m027 — 초기 카테고리 (예: 미수거, 누락, 가격불만, 응대불만 등)
- m028 — `none` 추가 (false positive unmark 용)
- m029 — pre/post 분리 (예약 전 불만 vs 예약 후 불만)
- m030 — 2단계 reclassify (Stage1 strict + Stage2 detail)
- m031 — Stage1 strict reclassify

### 운영팀 false positive unmark
- ComplaintModal 에서 "불만 아님" 버튼
- POST `/api/new_dashboard/complaints/unmark` → 해당 row 의 카테고리를 `none` 으로 변경
- 향후 재분류 시 운영팀 결정 보존

### 정확도 보장
- 초기 분류 후 운영팀이 검토 → false positive 누적
- m030/m031 의 reclassify 가 strict 패턴 적용 → 신규 데이터에 한해 정확도 향상

## CS Realtime 집계 로직

`/api/new_dashboard/cs-realtime/route.ts` 핵심:

### 데이터 소스
1. `messages` (방문) — assistant 24h
2. `lunch_messages` (런치) — assistant 24h
3. `channeltalk_reply_logs` (채널톡) — 24h
4. `app_settings` 의 `counselor:*` — 활성 상담사 명단
5. `cs_presence_log` — 오늘 KST 운영시간(10-22) heartbeat
6. 채널톡 `listAllUserChats({state:"opened"})` — 채널톡 큐깊이

### 집계
- **오늘 KST 하루 전체(00-24)** — 답변 수·세션 수 (운영시간 외 답변도 포함)
- **운영시간(10-22)** — 근무 분 (heartbeat) + FRT median
- **시스템별 분리** — todayBySystem.{visit, lunch, channeltalk}.{consults, replies}
- **lastReplyAt** — 24h 내 마지막 답변 시각 (외부 도구 답변자 active 표시용)

### 정렬
1. 답변 많은 순
2. 근무시간 긴 순

### 노이즈 제거
- `(미지정)`, `AI`, `시스템` — 카드 노출 제외

## 권한 가드

`ADMIN_DASHBOARD_ALLOWED_USERS` set 에 없는 사용자는 모든 `/api/new_dashboard/*` 가 403.

## 안전 장치

- DB 캐시 hit 률 모니터링 — Sonnet 호출 비용 폭주 방지
- 분류 한도: cron 1회 300건 → 부하 안정
- presence 채널 — 같은 이름 다중 탭 충돌 가능성 인지 (08-gotchas)
- 운영시간 외 heartbeat 는 cs_presence_log 에 들어가지만 카드 KPI 에는 안 잡힘 (정상)
