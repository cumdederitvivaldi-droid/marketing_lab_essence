# 03 — AI 로직 (Phase + `<order_data>` 파싱)

## 모델

| 용도 | 모델 | 위치 |
|---|---|---|
| 메인 응답 (Phase 4단계) | Anthropic Sonnet | `lib/ai/lunch-ai.ts` |
| 톤 보정 (선택) | Haiku | `/api/lunch/conversations/[sessionId]/polish/route.ts` |

방문수거와 마찬가지로 `app_settings.ai_provider` 로 런타임 전환.

## 4단계 Phase 머신 (경량)

```
idle (일반 안내)
  ↓ (주문 의사 감지)
order (정보 수집)
  ↓
confirm (주문 확정 + <order_data> 응답)
  
또는

idle
  ↓ (질의 감지)
inquiry (정산·일정·기타 질의응답)
```

방문수거 9단계 머신과 달리 **4단계로 간소화**. 런치는 정기 수거 + 단일 패턴이라 분기 적음.

**파일:**
- 메인: [`lib/ai/lunch-ai.ts`](../../../lib/ai/lunch-ai.ts)
- 프롬프트: [`lib/ai/lunch-prompt.ts`](../../../lib/ai/lunch-prompt.ts)
- 정책: [`lib/ai/lunch-policy.md`](../../../lib/ai/lunch-policy.md) — 빌드 타임 로드

## 톤 규칙 (방문수거와 다름)

| 규칙 | 이유 |
|---|---|
| **마크다운 금지** | 카카오톡에 그대로 출력되어야 함 |
| **이모지 금지** | 비즈니스 톤 |
| `:)` 만 허용 | 캐주얼 친근감 보존 |
| 존댓말 (방문과 동일) | 사장님 응대 |
| 짧고 명확 | 도시락 매장 사장님은 바쁨 |

`lib/ai/lunch-prompt.ts` 의 system 프롬프트가 위 규칙 명시 + 위반 시 자체 검수.

## `<order_data>` 자동 파싱

AI 가 주문 정보를 충분히 수집하면 응답 끝에 JSON 태그를 붙임:

```
사장님, 내일 도시락 50개 수거 등록해 드릴게요. 시간은 10시 30분 맞으시죠? :)

<order_data>
{
  "date": "2026-04-28",
  "pickup_time": "10:30",
  "box_count": "50",
  "vendor_name": "카페 ABC 강남점",
  "settlement_type": "link_pay",
  "notes": "현장 담당자 김매니저"
}
</order_data>
```

### 클라이언트 처리
1. `LunchChatView.tsx` 가 메시지 렌더 시 `<order_data>` 태그 검출
2. 사용자에게는 태그 제거된 본문만 표시
3. JSON 파싱 → "주문 등록" 버튼 활성화
4. 클릭 시 Orders 탭 신규 주문 모달 자동채움

### DB 저장
- `lunch_conversations.ai_order_data` (TEXT) 에 JSON 그대로 보관 (m017)
- 모달 닫기 후 재방문 시 복원용

### 실패 / 모호한 케이스
- 정보 부족 (날짜·시간·개수 중 하나라도 없음) → AI 가 재질문 (`order` phase 유지)
- 다중 주문 (예: "내일과 모레") → 한 번에 1건만 파싱, 나머지는 inquiry 로 안내

## 분류 (reply_kind) — 방문과 동일

`lib/utils/reply-classify.ts` 공유:
- `overlap ≥ 1.0` → ai_auto
- `overlap ≥ 0.6` → ai_assist
- 그 외 → human

저장: `lunch_messages.reply_kind` + `responded_in_ms` + `draft_char_overlap` (m026).

## Phase 전환 신호

`lib/ai/lunch-ai.ts` 가 user 메시지 분석 후 phase 결정:

| 신호 | 다음 Phase |
|---|---|
| 주문/수거/날짜·시간/개수 키워드 | `order` |
| 정산/세금계산서/매출 키워드 | `inquiry` |
| 일반 인사 / 잡담 | `idle` 유지 |
| `<order_data>` 응답 후 사장님 확인 | `confirm` |

## 메시지 직렬 (serial_number) — 중복 방지

`lunch_messages.serial_number` (m018, UNIQUE) — `${sessionId}_${serial}` 패턴.
- 해피톡 webhook 이 같은 메시지 중복 발송하는 케이스 방지
- INSERT 시 ON CONFLICT DO NOTHING

## RPC: `get_last_messages` (m015)

목록 화면(`/api/lunch/conversations`) 에서 N개 세션의 마지막 메시지 1건씩 한 번에 fetch.

```sql
SELECT * FROM get_last_messages(session_ids := ARRAY[...]::text[]);
```

서버 측 RPC 라 N+1 query 방지.

## 폴리시 문서 (lunch-policy.md)

`lib/ai/lunch-policy.md` 가 운영 정책 원문. 빌드 타임에 prompt 에 inline.

내용 (예시):
- 정기 수거 사이클 (주 N회 등)
- 정산 방식별 안내 멘트
- 거절·연기 응대 규칙
- 미수거·누락 사후 처리

수정 시 빌드 후 배포 필요.

## 안전 장치

- 마크다운/이모지 자가 검수: AI 가 자기 응답 검증 후 위반 시 재생성 (system prompt 명시)
- `<order_data>` JSON 파싱 실패: 클라이언트가 silently fail, 모달 자동채움 안 함
- Phase 강제 변경 라우트 없음 (방문과 달리) — 디버깅 시 직접 DB 수정
