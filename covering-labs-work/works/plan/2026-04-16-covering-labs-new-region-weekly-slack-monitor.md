# 신규 지역 주간 Slack 모니터 배치

> 유형: PRD | 플랜
> 작성일: 2026-04-16
> 상태: 완료

## 목표

- 신규 지역 운영 권역의 주간 성장 상태를 매주 Slack에서 한 번에 보고, `계속 밀 권역 / 곧 500 도달 권역 / 보류 권역`을 바로 판단할 수 있게 만든다.

## 현황 분석

- 신규 지역 숫자는 Grafana와 개별 분석 문서에 흩어져 있다.
- 현재 기준 운영 판단의 핵심은 `권역별 최근 30일 MAU`, `전주 대비 변화`, `500 도달 예상 시점`, `권역 매핑 누락 리스크`다.
- 기존 문서 검산 결과, 첫 구현은 `service_region_group` 기준 `91~98` 권역을 사용했다.
- 2026-04-27 후속 변경으로 천안/아산 분리와 대전 동구 포함이 필요해져, 현재 구현은 SQL에 명시한 행정구 묶음을 사용한다.

## 이번 작업 계약

- 이번에 끝낼 범위:
  - `covering-labs/apps/` 하위에 신규 지역 주간 Slack 배치 앱 1개를 만든다.
  - 초기 구현 기준인 운영 권역 `91~98`의 주간 집계 쿼리를 코드에 넣는다.
  - Slack 본문에 전체 요약, 권역별 점수판, 데이터 리스크, Grafana 링크를 넣는다.
  - 수동 실행으로 메시지 미리보기까지 확인한다.
- 이번에 하지 않을 범위:
  - Grafana 대시보드 수정
  - 권역 매핑 테이블 직접 수정
  - 광고 계정, 쿠폰 정책, 앱 배너 운영 변경
- 완료 기준:
  - 새 batch 앱이 로컬에서 실행되고,
  - 신규 지역 주간 요약 Slack 본문이 생성되며,
  - 권역별 `통과권 / 근접권 / 보류권`이 숫자와 함께 일관되게 나오면 완료다.
- 실제 확인 방법:
  - BigQuery 실쿼리 결과 확인
  - 배치 수동 실행 로그 확인
  - Slack 본문 dry-run 또는 미리보기 확인
- 주요 리스크:
  - `대전 동구` 권역 매핑 누락으로 strict 집계가 운영 해석보다 낮게 나올 수 있다.
  - 배포 환경에 Slack 토큰이 누락되면 자동 발송은 막힐 수 있다.

- 수동 검증 결과와 남은 리스크가 문서에 남아 있다.

## 후속 변경 2026-04-27

- 요청: Slack 모니터링의 지역 구분 값을 운영 그룹 ID가 아니라 명시된 행정구 묶음으로 정의한다.
- 변경:
  - 천안/아산은 `천안`과 `아산`으로 분리했다.
  - 대전/세종/청주는 `대전 유성`, `대전 서중`, `대전 대덕동`, `세종`, `청주 상당서원`, `청주 흥덕`, `청주 청원`으로 고정했다.
  - `NEW_REGION_GROUP_IDS` 기반 실행 옵션은 제거했다.
- 검증:
  - `python3 -m py_compile apps/private/new-region-weekly-monitor/src/main.py` 통과.
  - `python3 src/main.py --dry-run` 통과. Slack 발송은 하지 않았다.
  - 04/20~04/26 dry-run 기준: 총 MAU `1,838명`, 통과권 `1곳`, 근접권 `5곳`, 보류권 `3곳`.
  - 천안 `552명`, 아산 `109명`으로 분리 확인.
- 주의:
  - 기존 `천안 아산` 합산 MAU는 같은 사용자를 한 번만 세지만, 분리 후 전체 합산은 각 권역 MAU를 더하므로 두 지역을 모두 이용한 사용자가 중복 반영될 수 있다.
