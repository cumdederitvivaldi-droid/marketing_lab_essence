# 시작하기

> 대상: 모든 `@covering.app` 구성원
> 소요 시간: 5~10분 (처음 한 번)

---

## 1단계: gcloud CLI 설치

| OS | 설치 방법 |
|---|---|
| Mac | `brew install google-cloud-sdk` |
| Windows | https://cloud.google.com/sdk/docs/install 에서 설치파일 다운로드 |
| Linux | `curl https://sdk.cloud.google.com \| bash` |

## 2단계: 초기 설정

```bash
gcloud init
```

실행하면 브라우저가 열리며 Google 로그인을 요청합니다.

| 질문 | 답변 |
|---|---|
| 계정 선택 | `본인이름@covering.app` 선택 |
| 프로젝트 선택 | `covering-app-ccd23` 입력 |
| Zone 선택 | `asia-northeast3-a` 또는 asia-northeast 중 아무거나 (1번 "Do not set"도 무방) |

## 3단계: SSH 접속

```bash
gcloud compute ssh --zone "asia-northeast3-a" \
  "covering-labs-instance-20260306-050059" \
  --project "covering-app-ccd23"
```

처음 접속하면 SSH 키를 자동 생성합니다. passphrase는 비워도 됩니다 (그냥 Enter).

> **🔴 반드시 `@covering.app` 계정으로 로그인하세요.** 개인 Gmail 등 다른 계정으로는 접속 불가.

## 처음 접속하면 자동으로 세팅되는 것

1. 홈 디렉토리 + 짧은 이름 바로가기 생성 (`/home/이름`)
2. Cloud Storage 개인 폴더 생성 (`gs://covering-labs/이름/`)
3. gcloud CLI 설정 폴더 생성 (`~/snap/google-cloud-cli/`) — 서버에서 `gcloud` 또는 `gsutil`을 처음 실행할 때 자동 생성

별도 작업 필요 없이 자동으로 준비됩니다.

## 접속이 안 될 때

`compute.instances.get` 권한 오류가 나면:

**1. 프로젝트 확인**
```bash
gcloud config get-value project
# covering-app-ccd23 이어야 함
```

**2. 계정 확인**
```bash
gcloud auth list
# @covering.app 계정이 ACTIVE여야 함
```

**3. 재인증**
```bash
gcloud auth login
```

**4.** 그래도 안 되면 아래 "SSH 키 문제" 확인

### SSH 키 문제 (`Connection closed ... [preauth]`)

접속 시도 시 아무 에러 메시지 없이 바로 끊기거나, 터미널에 `Connection closed by ... [preauth]`가 표시되는 경우 — SSH 키가 GCP에 정상 등록되지 않은 것입니다.

**1. 계정 확인**
```bash
gcloud auth list
# @covering.app 계정이 ACTIVE인지 확인
# 개인 Gmail 등 다른 계정이 ACTIVE면 키가 등록되지 않음
```

ACTIVE 계정이 `@covering.app`이 아니면:
```bash
gcloud auth login
# → @covering.app 계정 선택
```

**2. 기존 SSH 키 정리**
```bash
rm -f ~/.ssh/google_compute_engine*
```

**3. SSH 재접속 (키 재생성 + 등록)**
```bash
gcloud compute ssh --zone "asia-northeast3-a" \
  "covering-labs-instance-20260306-050059" \
  --project "covering-app-ccd23"
```

passphrase를 물어보면 **그냥 Enter** (빈 값).

> 이 과정에서 `gcloud`가 새 SSH 키 쌍을 생성하고, 공개키를 GCP OS Login에 자동 등록합니다. 한 번 등록되면 이후에는 바로 접속됩니다.

**4. 호스트키 충돌 (`WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED`)**
```bash
# 기존 호스트키 삭제 (인스턴스 IP로 접속 시)
ssh-keygen -R 34.64.177.181

# 또는 known_hosts에서 직접 삭제
grep -n "34.64.177.181" ~/.ssh/known_hosts   # 줄 번호 확인
# 해당 줄 삭제 후 재접속
```

**5.** 위 방법을 모두 시도해도 안 되면 — `iam.serviceAccounts.actAs` 권한이 없을 수 있습니다.

이 인스턴스는 서비스 계정이 연결되어 있어서, SSH 접속에 `roles/compute.osLogin` **+ `roles/iam.serviceAccountUser`** 두 가지가 모두 필요합니다. 관리자(jun)에게 아래 확인을 요청하세요:

- SA 레벨에서 본인(또는 `all@covering.app`)에 `roles/iam.serviceAccountUser`가 부여되어 있는지
- 확인 방법: [04_권한과_보안.md](04_권한과_보안.md) > "SSH 접속과 서비스 계정" 참조

## 관리자 접속 (`dev@covering.app` 멤버만)

일반 접속 후 root로 전환:

```bash
sudo -i     # root 셸로 전환
exit        # 원래 계정으로 돌아감
```

> ⚠️ OS Login 환경에서는 `gcloud compute ssh root@...`으로 접속해도 개인 계정(`이름_covering_app`)으로 로그인됩니다. root 권한이 필요하면 반드시 `sudo -i`를 사용하세요.
