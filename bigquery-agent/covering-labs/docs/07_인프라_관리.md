## GitHub Actions 배포 SA 키 관리

> GitHub Actions가 VM에 배포할 때 사용하는 서비스 계정 키 관리

### GitHub Secret 등록 (최초 1회)

```bash
# 1. SA 키 생성
gcloud iam service-accounts keys create /tmp/sa-key.json \
  --iam-account=covering-labs-deploy@covering-app-ccd23.iam.gserviceaccount.com \
  --project=covering-app-ccd23

# 2. 키 내용 확인 후 GitHub Secret에 등록
#    covering-app/covering-labs 레포 →
#    Settings → Secrets → Actions → New repository secret
#    이름: GCP_SA_KEY  /  값: cat /tmp/sa-key.json 전체 내용

# 3. 로컬 키 파일 즉시 삭제 (보안)
rm /tmp/sa-key.json
```

### SA 키 교체 (유출/만료 시)

```bash
# 1. 기존 키 목록 확인

  --iam-account=covering-labs-deploy@covering-app-ccd23.iam.gserviceaccount.com

# 2. 기존 키 삭제

  --iam-account=covering-labs-deploy@covering-app-ccd23.iam.gserviceaccount.com

# 3. 신규 키 생성 후 GitHub Secret 업데이트 (위 절차 반복)
```

## 서비스 계정 actAs 권한 관리

인스턴스에 SA가 연결되어 있으므로, SSH 접속하려면 `roles/iam.serviceAccountUser`(= `actAs`)가 필요합니다. 현재 `all@covering.app`에 SA 레벨로 부여되어 있습니다.

```bash
# 현재 SA에 누가 actAs를 갖고 있는지 확인
gcloud iam service-accounts get-iam-policy \
  covering-labs@covering-app-ccd23.iam.gserviceaccount.com \
  --format="table(bindings.role, bindings.members)"

# 그룹/사용자에 actAs 부여 (SA 레벨 — 이 SA에만 적용)
gcloud iam service-accounts add-iam-policy-binding \
  covering-labs@covering-app-ccd23.iam.gserviceaccount.com \
  --member="group:all@covering.app" \
  --role="roles/iam.serviceAccountUser"

# actAs 제거
gcloud iam service-accounts remove-iam-policy-binding \
  covering-labs@covering-app-ccd23.iam.gserviceaccount.com \
  --member="user:사용자@covering.app" \
  --role="roles/iam.serviceAccountUser"
```

> **🔴 `actAs`를 프로젝트 레벨이 아닌 SA 레벨에 부여하세요.** 프로젝트 레벨로 부여하면 프로젝트 내 모든 SA에 대해 actAs가 허용됩니다.

> **콘솔에서 확인:** IAM & Admin → Service Accounts → `covering-labs@...` → Permissions 탭 → "PRINCIPALS WITH ACCESS TO THIS SERVICE ACCOUNT"

### Public VM 방화벽 설정 (HTTP/HTTPS 공개)

> **🔴 중요**: 방화벽 대상 태그는 반드시 `public-http-server` / `public-https-server` 전용 태그를 사용합니다.
> `http-server` / `https-server`는 private VM이 사용 중이므로 공유하면 private VM이 공개 노출됩니다.
> public VM 생성 시 `--tags=public-http-server,public-https-server`를 지정해야 합니다.

```bash
# HTTP/HTTPS를 0.0.0.0/0으로 공개 (public VM 전용 태그)
gcloud compute firewall-rules create public-labs-allow-http \
  --project=covering-app-ccd23 \
  --target-tags=public-http-server \
  --allow=tcp:80 \
  --source-ranges=0.0.0.0/0 \
  --description="Public labs VM HTTP access"

gcloud compute firewall-rules create public-labs-allow-https \
  --project=covering-app-ccd23 \
  --target-tags=public-https-server \
  --allow=tcp:443 \
  --source-ranges=0.0.0.0/0 \
  --description="Public labs VM HTTPS access"

# 기존 VM에 태그 추가/변경 (필요 시)
gcloud compute instances add-tags covering-labs-public \
  --zone=asia-northeast3-a --project=covering-app-ccd23 \
  --tags=public-http-server,public-https-server
```

### Public VM 초기 세팅 스크립트

VM 생성 후 SSH 접속하여 아래 스크립트를 실행하세요:

```bash
gcloud compute ssh covering-labs-public \
  --zone=asia-northeast3-a --project=covering-app-ccd23 \
  --command="$(cat <<'SETUP'
set -e

# 패키지 업데이트 및 설치
sudo apt-get update -qq
sudo apt-get install -y -qq nginx python3-pip python3-venv curl

# Node.js 20 설치
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

# PM2 설치
sudo npm install -g pm2

# /shared 디렉토리 구조 생성
sudo mkdir -p /shared/apps /shared/nginx-confs /shared/scripts
sudo touch /shared/port-registry.json && echo '{}' | sudo tee /shared/port-registry.json > /dev/null
sudo touch /shared/.env

# covering-dev 그룹 생성 + 권한 설정
# - 2775: setgid + group-writable (deploy SA가 covering-dev 그룹 멤버로서 /shared 쓰기 가능)
# - port-registry.json은 664 (그룹 쓰기 가능, deploy 중 업데이트됨)
# - .env는 640 (그룹 읽기만, 쓰기는 관리자만)
sudo groupadd covering-dev 2>/dev/null || true
SA_USER=$(whoami)
sudo usermod -aG covering-dev $SA_USER
sudo chown -R $SA_USER:covering-dev /shared
sudo find /shared -type d -exec chmod 2775 {} \;
sudo find /shared -type f ! -name '.env' -exec chmod 664 {} \;
sudo chmod 640 /shared/.env

# nginx sudoers 설정
echo '%covering-dev ALL=(ALL) NOPASSWD: /usr/sbin/nginx' | sudo tee /etc/sudoers.d/covering-labs-nginx
sudo chmod 440 /etc/sudoers.d/covering-labs-nginx

# nginx 설정
sudo tee /etc/nginx/sites-available/covering-labs > /dev/null <<'NGINX'
server {
    listen 80 default_server;
    listen [::]:80 default_server;
    server_name _;
    root /var/www/html;
    include /shared/nginx-confs/*.conf;
}
NGINX
sudo ln -sf /etc/nginx/sites-available/covering-labs /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

# /shared/.env 기본값 설정
echo 'SERVER_DOMAIN=public-labs.covering.app' | sudo tee -a /shared/.env

echo "=== Public VM 초기 세팅 완료 ==="
SETUP
)"
```

## 방화벽

| 규칙 이름 | 소스 | 포트 | 비고 |
|---|---|---|---|
| `default-allow-ssh` | `0.0.0.0/0` | 22 | 전체 개방 (OS Login이 인증) |
| `default-allow-http` | `43.200.63.250/32, 10.0.0.0/16, 10.2.0.0/22` | 80 | **VPN 전용** (2026-04-17 복구) |
| `default-allow-https` | `43.200.63.250/32, 10.0.0.0/16, 10.2.0.0/22` | 443 | **VPN 전용** (2026-04-17 복구) |
| `default-allow-internal` | `10.128.0.0/9` | 전체 | GCP 내부 |

> **변경 이력:**
> - 2026-04-14 이전: HTTP/HTTPS가 `10.2.0.0/22`, `43.200.63.250/32` 전용
> - 2026-04-14 ~ 2026-04-17: Let's Encrypt HTTP-01 갱신을 위해 `0.0.0.0/0`으로 임시 공개
> - **2026-04-17 저녁 (현재):** DNS-01(Route 53) 방식으로 전환 후 `43.200.63.250/32, 10.0.0.0/16, 10.2.0.0/22` VPN 전용으로 복구. L4(방화벽) + L7(nginx) 이중 방어.

### 방화벽 규칙 변경

```bash
# 현재 규칙 확인
gcloud compute firewall-rules list \
  --project=covering-app-ccd23 \
  --format="table(name,sourceRanges.list(),allowed[].map().firewall_rule().list())"

# HTTP/HTTPS를 VPN 전용으로 변경 (현재 설정값 — DNS-01 갱신이므로 방화벽과 무관)
gcloud compute firewall-rules update default-allow-http \
  --project=covering-app-ccd23 \
  --source-ranges=43.200.63.250/32,10.0.0.0/16,10.2.0.0/22

gcloud compute firewall-rules update default-allow-https \
  --project=covering-app-ccd23 \
  --source-ranges=43.200.63.250/32,10.0.0.0/16,10.2.0.0/22

# SSH도 VPN 전용으로 제한하려면 (선택)
gcloud compute firewall-rules update default-allow-ssh \
  --project=covering-app-ccd23 \
  --source-ranges=10.2.0.0/22
```

> ⚠️ SSH를 VPN 전용으로 제한하면 VPN 장애 시 긴급 접속 불가. HTTP/HTTPS는 DNS-01(Route 53) 방식으로 SSL이 갱신되므로 VPN 전용이어도 SSL 자동 갱신에 영향 없음.
