### SSH 접속과 서비스 계정 (`iam.serviceAccounts.actAs`)

이 인스턴스에는 서비스 계정이 연결되어 있습니다. GCP OS Login은 SSH 접속 시 **두 가지 권한을 모두 요구**합니다:

| 필요 권한 | 역할 | 용도 |
|---|---|---|
| `compute.instances.osLogin` | `roles/compute.osLogin` | SSH 접속 자체 |
| `iam.serviceAccounts.actAs` | `roles/iam.serviceAccountUser` | SA가 연결된 인스턴스에 접속 허가 |

둘 중 하나라도 없으면 `Connection closed ... [preauth]` 오류와 함께 접속이 거부됩니다.

현재 설정:
- `roles/compute.osLogin` → `all@covering.app` (프로젝트 IAM)
- `roles/iam.serviceAccountUser` → `all@covering.app` (**SA 레벨** — `covering-labs@...` SA에만 적용)

> **🟢 `actAs`는 위험하지 않나요?**
> SSH 접속이 되면 어차피 메타데이터 API로 SA 토큰을 가져올 수 있습니다 (아래 "인스턴스 메타데이터 API" 참조). `actAs`는 접속을 위한 관문일 뿐, 접속 후 할 수 있는 것의 범위는 변하지 않습니다. 단, SA에 강력한 권한(`roles/editor`, `objectAdmin` 등)이 부여되면 위험하므로 **SA 역할은 최소한으로 유지**해야 합니다.

## 보안 유의사항

### 🔴 인스턴스 메타데이터 API

인스턴스 안의 **모든 사용자**는 메타데이터 API를 통해 서비스 계정 토큰을 가져올 수 있습니다. 이것은 Linux 사용자 권한과 무관하게 GCP 인스턴스의 기본 동작입니다.

```bash
# 누구나 실행 가능
curl -H "Metadata-Flavor: Google" \
  http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
```

이 토큰으로 SA에 부여된 모든 권한을 사용할 수 있으므로, **SA 역할은 최소한으로 유지**해야 합니다.

현재 조치:
- Cloud Storage: `objectCreator` + `objectViewer` (업로드/읽기만, **삭제 불가**)
- BigQuery: `READER` (읽기만, 쓰기/삭제 불가)
- `objectAdmin` 사용 금지 (삭제 권한 포함)

### 🔴 Cloud Storage 파일 삭제

파일 삭제가 필요하면 **dev 팀이 `gsutil rm`을 sudo 없이 직접 실행** — 이것은 개인 인증(`gcloud auth login`)으로 동작하므로 SA 토큰과 무관.

### 🟡 AWS 내부 서비스 접근

VPN 경유로 AWS Private Subnet의 서비스에 접근 가능합니다. **모든 사용자**가 접근할 수 있으므로, 민감한 AWS 서비스가 있다면 AWS Security Group에서 GCP 대역(`10.178.0.0/20`)을 제한해야 합니다.

### 🟡 방화벽

| 포트 | 소스 | 비고 |
|---|---|---|
| SSH (22) | `0.0.0.0/0` | OS Login이 인증 담당. 필요 시 VPN으로 제한 가능 |
| HTTP (80) | `43.200.63.250/32, 10.0.0.0/16, 10.2.0.0/22` | **VPN 전용** — AWS Client VPN NAT + AWS VPC + 사내망만 허용 |
| HTTPS (443) | `43.200.63.250/32, 10.0.0.0/16, 10.2.0.0/22` | **VPN 전용** — 동일 |

#### Public VM (covering-labs-public) 방화벽

| 포트 | 소스 | 비고 |
|---|---|---|
| SSH (22) | `0.0.0.0/0` | OS Login이 인증 담당 |
| HTTP (80) | `0.0.0.0/0` | **전체 공개** — VPN 불필요 (target tag: `public-http-server`) |
| HTTPS (443) | `0.0.0.0/0` | **전체 공개** — VPN 불필요 (target tag: `public-https-server`) |

> public VM은 외부 공개용 앱 전용이므로 방화벽을 0.0.0.0/0으로 개방합니다.
> site-to-site VPN 미연결로 내부 AWS 리소스에 접근할 수 없습니다.
>
> **🔴 전용 태그 사용**: public VM은 반드시 `public-http-server` / `public-https-server` 태그를 붙여야 합니다.
> private VM의 공용 태그(`http-server` / `https-server`)를 공유하면 방화벽 규칙이 private VM에도 매칭되어 VPN 전용 보장이 깨집니다.

> **변경 이력**
> - 2026-04-14 이전: GCP 방화벽이 `10.2.0.0/22`, `43.200.63.250/32`(AWS Client VPN) 전용이었음.
> - 2026-04-14 ~ 2026-04-17 오후: Let's Encrypt 인증서 갱신을 위해 방화벽 `0.0.0.0/0` 공개, VPN 없이 모든 앱 접근 가능.
> - 2026-04-17 오후 (1차): **nginx `allow/deny`** 로 Layer 7 차단 도입. 방화벽은 `0.0.0.0/0` 유지.
> - **2026-04-17 저녁 (2차, 최종)**: Let's Encrypt 갱신을 HTTP-01 → **DNS-01 (Route 53 plugin)** 으로 전환 → **GCP 방화벽도 VPN IP 만 허용**하도록 복구. 이제 L4(방화벽) + L7(nginx) 이중 방어.
>
> **이중 방어 구조**
> - **L4 (GCP 방화벽)**: VPN NAT/VPC/사내망 CIDR 만 TCP 연결 허용. 그 외에는 TCP 핸드셰이크조차 안 됨 → timeout.
> - **L7 (nginx allow/deny)**: 방화벽을 통과한 요청이라도 `labs.covering.app` 서버 블록 및 default server 블록에서 동일 allowlist 로 재차 차단. 백업 방어.
>
> **Let's Encrypt 갱신 (DNS-01)**
> - certbot authenticator: `dns-route53` (snap plugin `certbot-dns-route53`)
> - AWS 자격증명: VM `/root/.aws/credentials` (mode 0600)
> - 권장: 전용 IAM user (최소 권한: `route53:ListHostedZones`, `route53:GetChange`, `route53:ChangeResourceRecordSets` on hostedzone `Z03245723K4D2WBHZVRID`) 키로 교체 필요
> - renewal config: `/etc/letsencrypt/renewal/labs.covering.app.conf` 의 `authenticator = dns-route53`, `installer = nginx` (cert 설치는 nginx 유지)
>
> **허용 경로**
> - AWS Client VPN (`covering-vpn-v2`) 연결 후 `https://labs.covering.app/[앱이름]`
> - IP 직접 접근(`http://34.64.177.181/*`)은 방화벽 및 nginx 양쪽에서 차단되거나 `Covering Labs` 기본 문구만 반환
> - nginx 설정 레퍼런스: `infra/nginx/labs.covering.app.conf`, `infra/nginx/covering-labs.conf`
